## ⚠ Open issue — under deployer review (2026-04-16)

**Finding 88 — `app/dedup_jobs.py` delete volume much larger than this doc
originally estimated.** The release note quoted "~357 excess rows" based on
the *800-row sample* in the Finding 88 report; the full-DB backlog is
materially larger (the sample's ~47% dup rate projects to tens of thousands
of rows across all 47,776 jobs).

Additional concern: the script collapses on exact `(company_id, title)`
equality. For Jobgether (the known aggregator that drove the finding) this
is correct. For non-aggregator companies that post the **same title in
different regions/teams** — e.g. "Software Engineer" at a global company
across US / EU / APAC offices — the script would incorrectly fold those
into one row.

**Status:** deployer is reviewing whether the applied delete scope is
correct before resuming the rest of the post-deploy runbook. Do **not**
re-run `app.dedup_jobs` (real run) until the review concludes. `--dry-run`
remains safe.

Diagnostic SQL — top 20 companies by excess-row count, run against the
current DB to see whether the pattern is Jobgether-only or includes
legitimate same-title-different-posting cases:

```sql
SELECT
  c.name AS company,
  COUNT(*) AS groups_with_dupes,
  SUM(dup.n - 1) AS excess_rows
FROM (
  SELECT company_id, title, COUNT(*) AS n
  FROM jobs
  GROUP BY company_id, title
  HAVING COUNT(*) > 1
) dup
JOIN companies c ON c.id = dup.company_id
GROUP BY c.name
ORDER BY excess_rows DESC
LIMIT 20;
```

If the top row is Jobgether with everyone else in single-digits → script
logic is fine, just a bigger cleanup than the doc implied; deployer can
proceed. If there are legitimate-looking companies with hundreds of
"duplicates" → the match key needs tightening (probably
`(company_id, title, location_raw)` or `(company_id, title, external_url)`)
before re-running. Fixer will follow up with a revised script if the
review concludes the second case.

Rollback of an already-committed delete run: the pre-deploy `pg_dump` in
`/opt/sales-platform/backups/pre-deploy-6e348a6-*.sql.gz` is the restore
source (7-day retention).

## Manual post-deploy steps

### 1. Job-duplication backlog cleanup (Finding 88, required)

The ingest guard in `scan_task.py` stops **new** dupes, but the ~47% backlog (Jobgether aggregator, up to 42 rows per logical role) is still in the DB. Run the one-shot cleanup.

```bash
ssh deploy@VM
cd /opt/sales-platform

# Dry-run first. Prints top-20 duplicate groups with company name
# and row counts. Touches nothing. Safe to run any time.
docker compose exec backend python -m app.dedup_jobs --dry-run

# Expected output shape:
#   Found N duplicate (company_id, title) groups; M rows across them
#   Top: 42× 'Jobgether' | 'Senior Designer (Brand, UI/UX)'
#        42× 'Jobgether' | 'Risk Operations Analyst'
#        ...

# If the sample looks right, apply. Batches DELETEs in chunks of 500
# IDs; survivor per group is MAX(first_seen_at), ties broken by MAX(id).
# Review + JobDescription cascade via ondelete=CASCADE /
# cascade="all, delete-orphan". Re-running after is a no-op.
docker compose exec backend python -m app.dedup_jobs
```

Expected post-run: "Total Jobs" on the Monitoring dashboard drops from ~47,776 → ~47,400 (Jobgether alone contributed ~357 excess rows). Per-platform row counts shift; the Lever source score distribution flattens. Review Queue stops repeating the same title 15× in a row.

### 2. Role-cluster retroactive rescore (Findings 91 / 92 / 93, optional)

The new negative-signal guards and keyword changes affect classification going forward. Already-stored `role_cluster` values on the 47k rows are stale until `rescore_jobs` runs.

Two options — either is fine.

**Option A — wait for the nightly Celery Beat schedule.** `rescore_jobs` runs at 03:00 UTC (see `celery_app.py`). Zero action required. Dashboard numbers settle overnight.

**Option B — trigger the task manually** if you want the cluster counts to update before the next review shift starts.

```bash

    app.workers.tasks.maintenance_task.rescore_jobs
```

Either way, some rows will flip state:
- `security` → unclassified for 60-70 legal/HR/clinical/pharma/tax compliance titles (Finding 91).
- `infra` → unclassified for 65-70 cloud-sales / cloud-marketing / hardware-systems titles (Finding 92).
- unclassified → `infra` for ~44 AWS/Azure/GCP titles that previously needed the `"aws engineer"` suffix (Finding 93).

Net: cluster counts on the Dashboard will **drop** (FPs outweigh FNs for the volume). That is the intended behavior; do not treat it as a regression.

## Rollback

`Actions → Rollback → Run workflow` with tag `sha-e495fa5`. <30 sec.

Specific concerns for rolling back this batch:

- **Alembic**: the `scoring_signals_user_scoping` migration is forward-compatible with the previous code (NULL rows are treated as shared). A rollback does **not** need a down-migration in normal operation. If you truly need to undo the schema: `docker compose exec backend alembic downgrade -1`.
- **`app/dedup_jobs.py` deletions**: not reversible from code. If step 1 was already applied and rollback is needed for an unrelated reason, the pre-deploy `pg_dump` in `/opt/sales-platform/backups/` is the restore source. `ci-deploy.sh:pre_deploy_backup` retains 7 days.
- **Ingest collapse**: rolling back `scan_task.py` restores the old "every Lever job-id is a new row" behavior. New duplicates will accumulate from the next scan onward; the deletions from step 1 remain removed.
